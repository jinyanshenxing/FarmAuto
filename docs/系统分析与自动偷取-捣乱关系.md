# 系统分析报告：漏洞与流畅性 · 自动偷取与捣乱关系

## 一、自动偷取与「自动捣乱」的关系（结论）

### 1. 执行顺序（已验证）

单次访问某位好友时，`visitFriendBody` 内的顺序为：

1. **帮助**：除草 → 除虫 → 浇水（若开启）
2. **偷菜**：可偷则先执行偷取（含 `stealDelaySeconds` 配置的延迟）
3. **捣乱**：放虫 → 放草（若开启）

因此：**在同一次访问中，偷菜始终在捣乱之前执行，自动偷取的速度不会因为开启「自动捣乱」而被延后。**

### 2. 对「速度」的两种理解

| 维度 | 是否受捣乱影响 | 说明 |
|------|----------------|------|
| **单好友内偷取时机** | **否** | 进入好友农场后先偷后捣乱，偷取不等待捣乱。 |
| **整轮巡田完成时间** | **是** | 捣乱使用 `putInsects`/`putWeeds`，每块地间隔 `sleep(100)`，会拉长该好友的停留时间，从而延后进入下一位好友，整轮巡田总时长增加。 |

### 3. 实现细节

- 偷菜：优先批量 `stealHarvest(gid, targetLands)`，失败则逐块重试，逐块间隔 100ms。
- 捣乱：`putInsects` / `putWeeds` 内部按地块循环，每块一次请求 + `sleep(100)`，无批量接口。
- 好友间间隔：每位好友访问后统一 `sleep(200)` 再访问下一位。

**建议**：若只关心「谁先被偷到、单次访问内先偷后捣乱」，当前逻辑已满足；若希望整轮巡田更快完成，可关闭自动捣乱或接受捣乱带来的额外耗时。代码中已加注释标明顺序，便于后续维护。

## 二、漏洞与安全排查

### 2.1 接口与权限

- **管理接口**：敏感操作（制卡、封禁、账号删除等）使用 `requireAdmin`，校验 `x-admin-token`。
- **账号隔离**：需账号维度的接口通过 `getAccIdOrRespond` 取 `x-account-id`，并经由 `resolveAccountId` / `canAccessAccount` 解析与鉴权，普通用户仅能操作本人账号。
- **配置与名单**：黑名单、农场封禁名单、设置等均按 `accountId` 读写，未发现跨账号越权读写。

### 2.2 输入与错误处理

- **账号/好友 ID**：`gid`、`accountId` 等在使用前做 `toNum`/存在性检查，缺失时返回 400 或安全降级。
- **封禁逻辑**：访问/操作好友时若返回信息包含「封禁」，会加入农场封禁名单并打日志，避免重复触发。
- **Worker 与 Store**：配置通过 `broadcastConfig` 同步到 worker，封禁名单的增删会持久化并生效。

### 2.3 前端与数据

- 列表与表格数据多为接口返回的只读展示，未发现将未转义 HTML 直接插入 DOM 的用法（Vue 默认转义）。
- 管理 token、账号 ID 通过 header 或前端存储传递，未在 URL 中暴露敏感参数。

### 2.4 建议

- 生产环境务必使用 HTTPS，避免 token 被窃听。
- 管理密码为服务端哈希存储，避免明文或弱哈希。
- 若存在未使用 `requireAdmin` 或未校验 `canAccessAccount` 的新接口，需补上鉴权与账号校验。

## 三、流畅性与性能

### 3.1 好友巡田

- **智能调度**：未成熟好友会按最早成熟时间延后访问，减少无效请求。
- **好友间间隔**：200ms，在流畅与风控之间折中。
- **单好友内**：帮助/偷菜/捣乱顺序固定；捣乱每块 100ms 会拉长该好友停留时间，见上文。

### 3.2 前端

- 状态与配置通过 Pinia、Socket 更新，避免频繁整页刷新。
- 列表与日志采用分页或虚拟滚动更佳（若数据量增大可再优化）。

### 3.3 后端

- Worker 按账号独立，互不阻塞。
- 配置与名单读写为内存 + 文件，无 DB 瓶颈；若账号数或名单很大，可再评估读写频率与格式。

---

## 四、小结

- **自动偷取与自动捣乱**：单次访问内**先偷后捣乱**，偷取速度**不受捣乱开关延后**；开启捣乱会**增加单好友停留时间和整轮巡田总时长**。已在 `friend.js` 中加注释标明顺序与影响。
- **安全**：管理接口与账号维度接口有鉴权与隔离，配置与名单按账号隔离；建议生产使用 HTTPS 并定期检查新接口权限。
- **流畅性**：巡田有智能调度与固定间隔，主要可感知延迟来自捣乱每块 100ms 及好友间 200ms；可按需关闭捣乱或后续优化捣乱间隔/批量以缩短整轮时间。
